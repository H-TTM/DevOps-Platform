Containers


Containers are a method of creating and running software applications so they are more lightweight and portable than traditional approaches. Containers bundle an application into a singular package which contains all its: dependencies, libraries, binaries, and configuration files, whilst also sharing the Operating System (OS) of the underlying infrastructure. Containerised applications are typically easier to manage, quick to start up, and have lower operational costs.

To provide a Platform of Understanding on Containers this blog will begin with a brief comparison of Containers and Virtual Machines (VM's), before providing an overview on Physical Infrastructure and VM's to highlight the demand for Containers. Lastly the blog will explain Containers, their benefits and disadvantages, and what they mean for organisations going forwards.

VM & Container Diagram

To understand what a Container is and how it works the below diagram which compares Containers and Virtual Machines can be a good place to start. This type of diagram is on most Container related blogs as it highlights how both are very similar but with a few key differences which significantly alter why and when they are used.

A brief summary on each of the elements in the diagram:

  • Infrastructure - The server/laptop/computer everything is running on
  • Host Operating System - The OS which exists on the Infrastructure (e.g. Windows/Linux).
  • Hypervisor - Software that is necessary for VM's to work
  • Container Daemon - Software that is necessary for Containers to work
  • Guest OS - OS of the Virtual Machine (notice how Containers don't have this)
  • Bins/Libs - Necessary Binaries & Libraries (code & files) for the application to work
  • App - Application

Side note: OS Kernel

An OS's kernel can be mentioned in other blogs, however it is not really necessary to know about it to understand Containers. However, if you're interested, a kernel is essentially the core part of an OS, all requests and actions pass through it.

Physical Infrastructure

Before VM's when a company created an application they would need to run it on infrastructure, such as server, that had sufficient capacity to meet the applications current and predicted future demand. As there would be no way to segregate the libraries and dependencies of the application from any other applications on the same infrastructure the company would either need to buy new infrastructure for each application or place similar applications together on the same infrastructure.

If a new server was purchased for each individual application it would end up costing the company a huge amount of money and if the application did not require all the servers capacity (they rarely did) then the company would have extremely inefficient and costly IT infrastructure.

If the company placed similar applications on the same server it would help overcome the wasted capacity issue, however as there is no way to segregate each applications libraries and dependencies the applications would often end up entwined over time which can result in the applications acting unpredictably. Additionally, it would be almost impossible to separate these applications without a huge amount of time, effort, and cost, to ensure there is no detrimental impact to the other applications on the shared infrastructure.

Virtualisation

The concept of VMs arrived in the 1960's and offered a unique approach to overcoming the limitations of physical hardware.

VMs enable multiple virtual servers with different OS's to run on a single physical server. VMs contain their own OS and all the binaries, libraries and dependencies of the applications running on them, providing a form of isolation for different applications on the physical infrastructure.

By being able to run more applications on infrastructure without OS limitations or applications becoming entwined it is possible to achieve significant savings in capital, operational, and energy costs.

Example: Mac vs Windows - Why choose?

Lets say you own a MacBook but want to use an application that only works on Windows 10. Once you've installed the necessary software to act as the Hypervisor on your MacBook you could create a VM with Windows 10 OS and run the Windows 10 specific applications on it. Additionally, if you were feeling nostalgic and wanted to use an application that only ran on Windows Vista you could create an additional VM on your MacBook with the Windows Vista OS and run the application on that VM. All that could be achieved on your MacBook, as long as it had enough space, and the Windows 10 and Windows 7 applications would be isolated from one another.

However, the key benefit of a VM, containing a full copy of its own OS, is also its main disadvantage. OS's require a lot of capacity and result in VM's typically taking up tens of GB's. When you have multiple VM's with the same OS it results in a significant amount of the infrastructures capacity being used unnecessary. Additionally, their size makes them slow to boot up, so applications within a VM can take several minutes to become available, which can be significant and costly.

Containers

Containerisation is not a new approach to creating and running applications, it can be even be traced back to the 1970's. However, the use of Containers has seen a massive increase in popularity since the emergence of new technologies and platforms (such as Docker) in 2013, which have dramatically increased the utility and capability of Containers.

A Container should only include a single application with all its necessary dependencies, libraries, binaries and configuration files. Containers isolate software from the infrastructure and other applications/Containers on the same infrastructure, making them more portable and easier to run on different infrastructure.

Containers enable the isolation of applications from one another even if they run on the same infrastructure (be it Virtual or Physical). Unlike VMs Containers do not include an OS so they are significantly smaller in size. Containers are typically tens of MB's instead of VM's which typically are tens of GB's. Consequentially, this enables Containers to be started up almost instantly, whereas VM's can take several minutes. This makes Containers a useful approach for increasing the management and scalability of applications which are developed in a Microservices approach (find out more on Microservices here).

Example: Why Containerise when you can Virtualise?

An organisation is developing multiple applications that all run on the same OS. They are not confident they can predict the capacity requirements of each applications future capacity requirements so want the applications to be easily portable between infrastructure.

If they decide to place all the applications on the same infrastructure there is a high likelihood that eventually the applications will become entwined, as mentioned in the Physical Infrastructure section above, making them extremely difficult if not impossible to migrate to different infrastructure.

If they place each application in different VM's they will increase the applications isolation so they are more portable however they will waste tens or hundreds of GB on the replication of each VM's OS.

However, if they develop each application within its own Container they can run the applications either directly on the Host Infrastructure or in a VM. This will avoid the large capacity requirements of VM's and each Container will be isolated from one another so they can easily be migrated to different infrastructure when they are required to.

It's important that the configuration and setup of Containers is done properly as they can pose a greater security risk than VMs. As Containers share the Host OS a single compromised container could affect the Host OS, which then could lead to the compromise of the entire machine and all its Containers. A compromised VM, on the other hand, would only affect that one VM and not affect the others. However, this risk is being reduced through new open source technologies.

Containers, the replacement for VM's?

Containers can be used by organisations to great effect to increase the utilisation of infrastructure and making the development and management of applications more Agile and efficient. But Containers do not remove the need to VM's, both VM's and Containers provide different capabilities which can help software delivery.

Containers should be seen as a new(ish) approach to creating and running applications which can be used to further the benefits of VM's or provide additional benefits in scenarios where VM's are not necessary.
