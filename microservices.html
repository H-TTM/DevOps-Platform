Microservices


Microservices are an approach to developing software applications which aims to segregate different functionalities into single-function independent services. Microservices perform standalone activities which typically communicate with one another through API's (Application Programming Interfaces). This has grown in popularity through Agile and DevOps (more on these can be found here) as Microservices can enable faster, more efficient software delivery.

To provide a Platform of Understanding on Microservices this blog will first explain the traditional approach to developing software before moving onto Microservices, using an example throughout to highlight how these approaches differ and the benefits and disadvantages of each. The blog will end with a view on which approach is better suited for organisations.

Side note: Application Programming Interface (API)

An API at its core is a mechanism which enables applications to communicate and interact with other applications, sending specific information, or actioning simple commands.

A way to visualise this is by looking at how phones and Smart Watches (not the ones which have their own SIM) work together. When a text is received by the phone it is sent to the Smart Watch where it can be read, and replies can be sent via the watch through the phone. The watch is not receiving or sending the text by itself, however the API's in both the phone and watch are able to communicate through the sharing of information and commands. Although there is a lot more going on that what was just discussed, this example gives an understanding of how API's work together.

'Monoliths' - Traditional approach

'Monolith' is a name provided to how software services/applications have traditionally, and still continue to be, developed. This is where an entire service is built as a single block of code (a code base), which contains all the functions and business activities the application performs.

Containing everything needed for a service within the same code base can be seen as a logical approach as it should make it easier to manage. However, this approach can quickly become problematic.

To better explain how a Monolith is formed and the issues that are commonly encountered the below example will be used, with a rather unimaginatively named, Company A.


Example - Part 1: Monolith - Single application service

Company A wants to create a 'Welcome Email' service which obtains a new users contact information from a database, forms a preset standardised welcome email, and sends it to them. A simple and effective service as shown below.


Example - Part 2: Monolith - Multiple application service

Company A then want to also create:

  • 'Promotion Email' service to send promotional emails to its customers
  • 'Cancel Email' service to send a confirmation email when a customers request to cancel their subscription
Both of these additional services would contain much of the same functionality as the ‘Welcome Email’ service as both would require retrieving customer contact details from the database and sending a standardised email to the customer. In addition, these two new services would also need to retrieve information on how long the user has been a customer to include in the standardised emails.

As the 'Welcome Email' service was created under one single code base the new 'Promotion Email' and 'Cancel Email' services have 2 options; either duplicate the ‘Welcome Email’ service code and adapt it slightly to achieve the service requirements or the alternative is to combine all three email services together in the one code base.
If they decided to avoid the duplication of code and opt for the second option outlined above, without tailoring it would result in a Welcome Email being sent alongside a Cancel or Promotion email. This is due to the 'Welcome Email' code being contained in one code base which stops certain elements of the code being used in isolation.

The decision is made to avoid duplication of effort and code by combing all three into a 'Email' service. So now the 'Email' service contains:

  • 'Welcome Email' service
  • 'Promotion Email' service
  • 'Cancel Email' service

This approach now means that additional decision points need to be added into the code to ensure that the correct email is sent to the correct customer. The 'Email' service now looks like the below diagram.

Pitfalls of traditional 'Monolith' approach

Containing all the code for multiple services under one code base provides the benefit of reducing duplication of code. However, as more and more services and functionalities are added under one code base the more complicated it gets to manage. Any alteration to any of the services included in the code base need to ensure there is no detrimental impact to any of the other services included in the code base. This results in changes taking longer than necessary as significant amounts of testing needs to be carried out for any change, big or small.

Additionally, the more code that exists in one code base will result in a decrease in its overall performance, as code needs to be added to ensure that when one service is called it doesn’t accidentally result in another service in the code base being called.

In terms of the Company A example, the performance of the simple 'Welcome Email' service will now be considerably worse as additional steps have been added into the code base to ensure not all 3 emails are sent each time the Monolithic 'Email' service is called.

Microservices

Microservices, or Microservice Architecture aims to overcome the challenges of Monolithic services by taking the complete opposite approach to code management. The aim is to split code into segregated, independent services, which perform isolated actions in their simplest form.

3 key characteristics of Microservices:

  • Can be written in different programming languages
  • Highly maintainable and testable (can be created, updated, or deployed without impacting other Microservices)
  • Each performs its own function, based around business capabilities

As Microservices can be written in different languages and still work with each other they enable a completely different way of operating than under the traditional approach.

Traditionally, teams will make updates to 'Monoliths', sometimes simultaneously, adding desired features. This can be slow and painful as updates to the code might not follow the same coding standards which can result in nonsensical code descriptions or varying quality, amongst other things. This is made worse when another teams code needs to be fixed due to it causing an update not to work.

Whereas Microservices can be split into groups of similar functionality and ownership can be maintained by small teams. This ensures consistency within the grouped Microservices as ownership is relatively static and any updates are fully understood before implementation, and are implemented with the same code quality, which helps to avoid any unnecessary or detrimental code being added.

Example - Part 3: Monolith to Microservices

Company A decided to take the Microservices approach instead of the Monolithic approach to developing their applications. As a result, they have broken down the 'Email' services into small pieces of functionality and developed as seperate Microservices. By splitting the services into individual functionalities within their own code bases it has removed the necessity to combine code to utilise desired functionalities. This now enables all Email services to remove unnecessary actions and decision points whilst still utilising shared functionality, increasing their performance and making them easier to manage.

So, Microservices solve everything right?

Well, not exactly. It's not realistic to claim Microservices are a perfect approach to creating services as they too come with their own complications that if not handled can be extremely problematic. 2 key areas are:

  • Complexity: As the ancient proverb goes - Would you rather fight one horse-sized duck, or a hundred duck-sized horses? Microservices are smaller and have numerous interconnections and interdependencies. As a result, it requires a firm architectural overview from the beginning to ensure clarity on how they work with one another
  • Sizing: Microservices need to be able to handle tasks by themselves but not contain so much logic that they become bottlenecks. If they are made too small then it increases the burden on dependency mapping and maintenance

Microservices vs 'Monoliths'

Microservices go hand-in-hand with Agile and DevOps orientated organisations. Microservices can be leveraged to great effect for building and maintaining efficient and effective Enterprise applications as they can be created, updated, and deployed with little or no impact to other Microservices.

It is not right to suggest organisations should stop working on Monolithic applications and re-architect them into Microservices, as both approaches provide their own benefits and disadvantages. Also, Monolithic applications might require significant effort, both time and cost, to re-architect and provide little value.

However, as organisations increase their DevOps maturity Microservices should be used for new applications and functions to capitalise on the benefits they can bring. Additionally, the rising capability of new tools and technologies can be used to reduce or overcome the complexities of Microservices, helping to further their benefits.

Companies that have been able to harness the benefits of Microservices in their DevOps maturity include: Amazon, Netflix & PayPal and as a result they have been able to accomplish delivery capabilities that would have been seen as impossible a few years ago, such as being able to deploy new changes into Production hundreds and even thousands of times a day.
